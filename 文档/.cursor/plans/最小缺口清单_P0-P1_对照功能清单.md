# 最小缺口清单（P0/P1）——对照《功能清单》

更新时间：2026-04-21  
目标：以论文/答辩“可验收闭环”为准，补齐**最少**功能，使系统从“核心主线可用”提升到“功能清单关键点可演示”。

> 说明：当前项目已具备核心主线：登录/注册（基础）→ GitHub 绑定 → 信用评估（V2）→ 协作发布/申请/审核/互评 → 通知（含 WebSocket）→ 个人中心（基础+部分超集）→ 管理后台（多项）。  
> 但《功能清单》包含大量“扩展/规划级”项（私信、群聊、Redis、备份恢复等），不可能用最小改动一次性全部实现。

---

## P0（必须补齐，否则与文档差距过大/答辩容易被问住）

### P0-1 多方式登录：GitHub 第三方授权登录“可用且闭环”
- **文档条目**：一（二）2 多方式登录（含 GitHub 第三方授权登录）
- **现状**：已有 `/auth/callback` + 后端 `/api/user/oauth/login`；但需确保“未登录 OAuth 登录”在演示环境稳定、错误提示清晰。
- **验收标准**：
  - 未登录时点击 GitHub 登录 → 回调成功 → 拿到 token → 自动进入首页/重定向页
  - 回调失败（code 缺失/state 不匹配/后端拒绝）提示明确且可重试
- **涉及代码**：
  - 前端：`front/front-spa/src/ui/views/LoginPage.vue`、`AuthCallbackPage.vue`、`oauthGithub.ts`
  - 后端：`UserController.oauthLogin`

### P0-3 账号注册：补齐“邮箱注册”或将文档表述落地为“手机号/邮箱二选一”
- **文档条目**：一（一）1 账号注册：支持手机号/邮箱注册
- **现状**：注册页强制手机号；后端 `UserService.register` 支持 phone/email 但前端未提供 email 输入。
- **最小实现路径（二选一）**：
  - **A（推荐）**：注册页增加 email 输入，允许 phone/email 至少填一项；后端保持唯一性校验
  - **B**：若论文允许修改文档：将“手机号/邮箱注册”改成“手机号注册（可选绑定邮箱）”
- **验收标准**：
  - 能用邮箱完成注册并登录（或文档同步修改为一致口径）

### P0-4 社交私信（文档强项之一）：做“最小可用”
- **文档条目**：一（三）7 私信功能：一对一私信，支持文字/代码片段/链接
- **现状**：无接口/表/页面。
- **最小实现建议**：
  - 后端：建表 `dm_message`（fromUserId、toUserId、content、createTime、isRead）
  - API：`/api/dm/send`、`/api/dm/list?withUserId=`、`/api/dm/conversations`
  - 推送：复用现有 `/ws` 用户队列推送事件（新 event：`DM` 或复用 `SYSTEM`）
  - 前端：在 `/u/:username` 增加“私信”按钮；新增 `/dm` 页面显示会话与消息
- **验收标准**：
  - A 给 B 发消息 → B 通知中心/私信页可见 → 双向对话可继续

## P1（强烈建议补齐：提升“功能清单一致性”和答辩观感）

### P1-1 隐私设置（最小字段 + UI）
- **文档条目**：一（一）3 隐私设置（信用公开范围/动态可见性/消息接收权限）
- **现状**：无字段/页面。
- **最小实现建议**：
  - user 表新增：`privacy_credit_public`、`privacy_feed_public`、`privacy_allow_message`
  - `/api/user/privacy/get`、`/api/user/privacy/save`
  - 前端：`/me` 的“安全/设置” Tab 增加开关
- **验收标准**：开关可保存并影响对应页面展示（至少信用明细/主页展示能体现差异）

### P1-2 个人数据中心导出（最小：信用历史/协作参与/社交记录）
- **文档条目**：一（一）5 数据导出
- **现状**：信用导出已有；但个人中心“统一导出”没有。
- **最小实现建议**：
  - 后端：新增 `/api/me/export` 输出 CSV/ZIP（信用历史、我的发布/申请、关注/粉丝摘要、反馈摘要）
  - 前端：`/me` 增加“导出我的数据”按钮

### P1-3 分类信用排行榜“专项榜口径”对齐
- **文档条目**：一（二）3 多维度榜单（前端/后端/全栈/专项榜等）
- **现状**：`/api/credit/rank2` 支持 tag/level/分数区间/站内用户筛选，但没有“前端/后端/全栈”等预设榜单。
- **最小实现建议**：
  - 在前端提供“预设筛选模板”：如“前端=tag 包含 Vue/React/TS”等；本质仍调用 `rank2`
  - 或后端增加 `preset` 参数映射到 tag 组合（更规范）

### P1-4 社群“群聊”不做全 IM，但做“公告+文件分享+成员信用查看”闭环
- **文档条目**：一（三）4 社群互动（群聊/文件分享/成员信用等级查看）
- **现状**：社群已有部分；“群聊/文件分享”缺口大。
- **最小实现建议**：
  - 群聊改为“帖子式群内讨论”（不做实时 IM），文件作为附件上传到 `uploads/community`
  - 成员信用等级：直接读 `credit_score` 快照展示

## 明确“不纳入最小清单”的项（建议在论文中改成“展望”）
- Redis 缓存、分布式存储、数据备份恢复、系统监控大盘、协同过滤推荐等：实现成本高，且不影响核心闭环演示。

---

## 建议实施顺序（最小投入最大收益）
1. **P0-3 邮箱注册/文档口径统一**
2. **P0-4 私信（最小可用 + 通知推送）**
3. P1-1 隐私设置
4. P1-2 个人数据导出
