# 开源开发者协作与信用评估平台 — 项目整体介绍

> 本文档面向论文答辩、开题/中期说明、对外介绍与团队协作，概括项目定位、价值、用户群体、已实现能力与技术栈。后端工程名为 **`dev-platform-backend`**，前端以静态页面调用 REST API 的方式联调（具体页面以实际仓库为准）。

---

## 一、项目概述

本项目是一个面向开源生态与开发者社区的 **Web 应用系统**，在统一平台内提供：

- 基于 **GitHub 公开数据** 的开发者 **多场景信用评估**（综合维度得分、等级、历史趋势与可视化洞察）；
- **开发者社交与内容**：动态帖子、点赞/评论/收藏、关注与关注流；
- **协作撮合**：协作需求发布、申请与审核、与信用门槛结合；
- **社群组织**：社群创建、入群申请与审核、成员与社群内内容；
- **发现与运营**：全局搜索、首页聚合推荐、站内消息、意见反馈、帮助中心与管理后台。

技术形态上采用 **前后端分离**：后端为 **Spring Boot + MySQL + MyBatis-Plus** 的 RESTful API；前端为 **HTML/CSS/JavaScript** 静态站点，通过 **JWT** 与后端交互。系统强调 **可本地一键运行**（含数据库表结构自动兼容补齐），适合毕业设计与课程大作业场景。

## 二、建设背景与要解决的问题

开源协作高度依赖信任，但参与者往往难以在短时间判断合作方的 **工程习惯、协作意愿与历史表现**。与此同时，技术社区平台（如技术博客站、代码托管站）功能重心不同：前者偏内容阅读，后者偏代码与 PR/Issue，**缺少将「可量化的公开贡献画像」与「站内找人、发帖、做项目」串在同一条产品链上**的轻量平台。

本项目试图在 **不替代 Git 托管** 的前提下，解决以下问题：

1. **信息分散**：GitHub 账号背后是谁、是否适合参与某个协作任务，缺少与站内身份、标签、行为结合的展示。
2. **协作撮合成本高**：发布需求方难以快速筛出「标签匹配且信用达标」的开发者。
3. **社区冷启动**：仅有论坛式发帖不足以形成「评估—关注—协作—社群」闭环。

## 三、项目定位

| 维度 | 定位说明 |
|------|----------|
| **产品类型** | 开发者社区 + 信用画像 + 协作/社群撮合的 **垂直社区平台**（非代码托管、非全文技术媒体站）。 |
| **与 GitHub 的关系** | GitHub 作为 **权威公开数据源** 与 OAuth 身份来源之一；本平台侧重 **站内关系链、需求与社群治理**，不实现 Git、PR、Actions 等底层能力。 |
| **与 CSDN 等的关系** | 可选聚合或引用站外技术热点（规划中，见《新增功能计划》）；核心内容仍以 **站内 UGC（帖子、协作、社群）** 为主。 |
| **差异化** | **多场景信用分**（可配置权重）、**查询历史与对比/排行**、**协作最低信用门槛**、**管理端配置与审核**，形成「数据—信任—行为」闭环叙事。 |

## 四、目标用户群体

1. **高校学生与初阶开发者**  
   需要展示个人技术标签、参与协作练手、在社群中找到学习伙伴；可通过绑定 GitHub 获得可视化「信用画像」辅助自我总结（非唯一评价标准）。

2. **小型开源项目维护者 / 课题负责人**  
   发布协作需求，希望申请者满足一定信用门槛，降低沟通成本。

3. **技术社群组织者**  
   创建社群、审核入群、发布公告类内容，与站内动态形成互补。

4. **平台运营与管理员**  
   通过管理后台维护用户状态、内容举报、系统配置与信用权重，保障演示与实验环境可控。

## 五、项目价值

### 5.1 对用户与业务场景的价值

- **降低协作试错成本**：用可解释的维度分与等级，为「是否邀请某人参与任务」提供参考（平台需在文案中声明：信用分为模型估算，不构成就业或法律担保）。
- **提升社区活跃度**：关注流、互动、社群与搜索形成「发现—互动—沉淀」路径。
- **可演示、可度量**：首页汇总、排行榜、趋势与洞察接口便于答辩演示与论文中的「系统实现」章节截图。

### 5.2 对教学与科研的参考价值

- 覆盖 **Spring 生态、REST API 设计、JWT 鉴权、第三方 OAuth、定时任务、关系型数据建模** 等典型课程知识点。
- 信用模块可延伸讨论 **可解释性、公平性、数据时效、反作弊** 等议题，适合写在「总结与展望」中。

## 六、核心业务与已实现功能（按模块）

以下以后端 **`dev-platform-backend`** 已暴露能力为主归纳；若前端某页未接入，以接口为准。

### 6.1 用户与账号

- 用户注册、密码登录、个人信息查询与更新（含头像、昵称、简介、技术标签等扩展字段，以库表为准）。
- **GitHub OAuth** 登录与绑定流程（依赖配置的 `client-id` / `client-secret` / `token` 等）。
- 修改密码、账号状态（含管理端冻结/解封等能力，以 `AdminController` 为准）。

### 6.2 信用评估

- 按 GitHub 用户名进行 **单用户信用查询**，支持多 **评估场景**（与场景权重配置联动）。
- **排行榜**：基础排行与带标签/等级/分数区间的进阶排行。
- **多用户对比**（人数上限可配置，如 2～5 人）。
- **信用历史趋势**（按月份等粒度）、**洞察类数据**（供雷达图/占比等前端展示）。
- **查询历史**：登录用户可查；支持删除、批量删除、收藏某条历史（防越权）。
- **CSV 导出**等报告类能力（以接口实现为准）。
- **定时任务**：例如对档案库中账号的周期性刷新（具体 Cron 以代码为准）。

### 6.3 开发者社交与内容

- **帖子**：发布、列表、详情、点赞、评论、收藏、删除（含帖子状态如待审/已通过等字段时与管理端联动）。
- **关注 / 粉丝** 与 **关注动态流**（分页）。
- **个人主页聚合**：用户名维度下资料、信用、近期帖子等（以 `ProfileController` 实现为准）。

### 6.4 协作与申请

- 协作需求发布、列表、详情、申请、审核（申请状态与协作状态以业务代码为准）。
- 与 **最低信用分** 等业务规则结合（具体字段名见实体 `Collaboration` 与库表）。

### 6.5 社群

- 社群创建、列表与详情、按标签筛选。
- 入群申请、审核、成员列表、社群内发帖/公告类内容（以 `CommunityController` / `CommunityService` 为准）。

### 6.6 发现、首页与搜索

- **首页**：统计摘要、推荐协作、最新动态、热门开发者、热门帖子等。
- **全局搜索**：开发者、协作、帖子、社群等类型（关键词与部分筛选条件）。

### 6.7 收藏与统一兴趣表达

- 统一收藏接口：支持多种收藏类型（如 GitHub 用户、平台用户、信用报告、社群等，以 `FavoriteController` 文档与实现为准）。

### 6.8 消息、反馈与帮助

- **站内消息**：列表、已读、全部已读、删除、清空等（以 `MessageController` 为准）。
- **意见反馈**：提交与管理员查看、回复（可与站内通知联动）。
- **帮助中心**：配置化 Markdown 内容（如 `system_config` 存储，由接口返回）。

### 6.9 管理后台

- 管理员鉴权（JWT 内角色区分，与拦截器/配置一致）。
- 仪表盘、用户列表与状态管理、反馈处理、举报处理、帖子/内容审核相关能力。
- **系统配置**与 **信用场景权重** 维护；查询日志、管理员操作日志等运维向接口。
- 部分场景下 **手动触发信用重算** 等运维能力（以实际接口为准）。

### 6.10 工程化与可维护性

- **启动时数据库结构兼容**：对缺失表/字段/索引进行补齐（`DatabaseSchemaInitializer`），降低环境搭建成本。
- **请求追踪**：如 traceId 注入日志，便于排查（以 `RequestTraceFilter` 等为准）。
- **统一响应体与全局异常处理**（`Result`、`GlobalExceptionHandler` 等）。

### 6.11 规划中 / 文档级能力

部分能力已在《新增功能计划_体验与闭环增强》等文档中描述（如站外热门聚合、徽章、@ 提及等），**是否已全部落地以前后端仓库与接口清单为准**；论文中建议区分「已实现」与「展望」。

## 七、技术栈与架构说明

### 7.1 后端技术栈

| 类别 | 技术选型 | 说明 |
|------|----------|------|
| 语言与运行环境 | **Java 8** | 与 Spring Boot 2.7 及校方环境常见要求兼容。 |
| 框架 | **Spring Boot 2.7.x**、**Spring Web**、**Spring Scheduling** | REST、定时任务、IoC 容器。 |
| 安全 | **Spring Security**、**JJWT** | 安全链与 JWT 签发/校验；部分接口放行、部分需登录或管理员角色。 |
| 持久层 | **MyBatis-Plus 3.5.x**、**MySQL**（`mysql-connector-j`） | ORM 与 SQL 映射；部分统计/搜索使用 **JdbcTemplate**。 |
| 工具 | **Lombok** | 简化实体与 VO 样板代码。 |
| HTTP 客户端 | **RestTemplate** | 调用 GitHub OAuth、GitHub REST API 等；支持可配置 HTTP 代理（便于校园网环境）。 |
| 构建 | **Maven** | 依赖管理与打包。 |

### 7.2 前端与交互

- **静态页面**：HTML + CSS + JavaScript，通过 `fetch` / `axios` 等调用后端 API。
- **跨域**：后端对前端来源配置 **CORS**（开发环境常见为放开或指定本地文件/端口）。
- **身份携带**：登录后 **JWT** 存于 `localStorage` 或内存，请求头 `Authorization: Bearer <token>`（以前端实现为准）。

### 7.3 第三方与外部服务

- **GitHub**：OAuth 与 REST API（用户公开信息、仓库列表等，用于信用评估与绑定）。
- 其他站外数据源（如技术热榜）若在计划中，应单独说明 **合规、频率与降级策略**。

### 7.4 逻辑架构（文字版）

```
浏览器（静态前端）
    │  HTTPS/HTTP + JSON + JWT
    ▼
Spring Boot（Controller → Service → Mapper / JdbcTemplate）
    │
    ├── MySQL（业务数据、信用结果、日志、配置）
    └── GitHub API（采集与 OAuth）
```

## 八、非功能特性简述

- **安全**：密码存储需符合业界实践（如摘要算法，以 `UserService` 为准）；敏感配置走环境变量或配置文件，**勿将 token 提交到公开仓库**。
- **权限**：普通用户与管理员角色分离；涉及「本人数据」的接口需 **用户 ID 与 token 一致** 防越权。
- **可用性**：外部 API 失败时应有明确错误信息与日志；信用计算可考虑超时与降级提示。
- **可扩展性**：权重与系统配置入库，便于调参；模块化包结构便于增加新 Controller/Service。
